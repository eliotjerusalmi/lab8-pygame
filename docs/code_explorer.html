📋 Project Overview

A dynamic Pygame simulation featuring animated squares with intelligent collision avoidance and fleeing behavior. Smaller squares flee from larger ones, creating an emergent ecosystem behavior.

🎯 Key Features

  • Movement Squares move with physics-based velocity
  • Collision Borders bounce squares back dynamically
  • Fleeing Intelligent distance-based avoidance behavior
  • Lifecycle Squares spawn and expire over time
  • Performance FPS display and delta-time calculations

Key Insight:

Smaller squares move faster than larger ones, creating a natural feel to the simulation.

➡️ Movement & Physics

set_random_direction()

Assigns a random velocity vector with consistent speed:

angle = random.uniform(0, 2π) vx = cos(angle) × speed vy = sin(angle) × speed

move(delta_time)

Updates position and handles wall bouncing:

x += vx × delta_time y += vy × delta_time if x ≤ 0 or x + size ≥ WIDTH: vx *= -1 # reverse direction if y ≤ 0 or y + size ≥ HEIGHT: vy *= -1 # reverse direction

Note: Delta-time ensures frame-rate independence.

🏃 Intelligent Fleeing Behavior

The most sophisticated part of the simulation. Smaller squares detect larger squares and flee from them within a defined distance.

⚡ Algorithm Overview

larger?} D -->|No| C D -->|Yes| E{Distance
< 120px?} E -->|No| C E -->|Yes| F["Calculate repulsion"] F --> G["Add random noise"] G --> H["Normalize vector"] H --> I["Apply to velocity"] I --> J{More
others?} J -->|Yes| C J -->|No| K{More

🧮 Mathematical Details

Repulsion Vector Calculation:

dx = smaller.x - larger.x dy = smaller.y - larger.y distance = √(dx² + dy²) if 0 < distance < FLEE_DISTANCE: strength = (FLEE_DISTANCE - distance) / FLEE_DISTANCE flee_x += (dx / distance) × strength flee_y += (dy / distance) × strength

The strength decreases linearly with distance, creating natural behavior.

Adding Randomness:

random_angle = random.uniform(0, 2π) rand_x = cos(random_angle) rand_y = sin(random_angle) final_x = flee_x + rand_x × 0.35 final_y = flee_y + rand_y × 0.35

The constant 0.35 (RANDOM_DIRECTION_STRENGTH) balances deterministic and chaotic behavior.

🎯 Behavior Characteristics

  • Selective Only smaller squares flee
  • Reactive Only within 120px range
  • Stochastic Randomness prevents perfect patterns
  • Smooth Gradual velocity changes

💡 Key Insights

  • The FLEE_DISTANCE parameter creates the "sensing radius" for threat detection
  • Multiple larger squares create a combined repulsion effect (vector sum)
  • The randomness prevents emergent cyclic patterns
  • This is called a boid-like behavior in game dev

🔧 Tuning Guide

  • More chaotic behavior: Increase RANDOM_DIRECTION_STRENGTH
  • More squares: Increase NUM_SQUARES (impacts performance)
  • Longer direction changes: Increase DIRECTION_CHANGE_INTERVAL
  • More aggressive fleeing: Decrease FLEE_DISTANCE or increase RANDOM_DIRECTION_STRENGTH
  • Longer square lives: Increase MAX_LIFESPAN_MS

Architecture

The project is a compact game loop built around one entity class. The diagrams below trace control flow, dependencies, runtime timing, and the data that moves through the frame.

Call Graph
`main()` orchestrates the frame loop, and the `Square` methods split movement, drawing, and group behavior into clear units.
Dependency Graph
The project depends on three standard libraries plus Pygame, which owns the window, timing, events, drawing, and font APIs.
Full Sequence
A single frame starts with the clock tick, checks events, refreshes behavior, updates positions, and ends by drawing and flipping the screen.
Fleeing Sequence
The fleeing logic measures distance to larger squares, combines the strongest escape direction with randomness, and turns that into new velocity.
Motion Sequence
Movement uses the current velocity to update position, then clamps the square to the window and reflects the velocity when a wall is hit.
Data Flow
The main data paths are the square list, frame delta time, flee force, and the timer that triggers direction resets.
Good Patterns
✅Encapsulated square behavior
`Square` keeps position, velocity, color, size, and rendering in one object, which is the right boundary for a moving entity.
Basics:The object stores the data and knows how to draw and move itself.
Engineering Insight:This keeps the frame loop from turning into a long block of square-specific logic.
Architecture Insight:The class acts as a reusable entity type for future game objects.
✅Frame-rate independent motion
The code multiplies velocity by `delta_time`, so movement stays stable when the frame rate changes.
Basics:Shorter or longer frames still produce the same general distance over time.
Engineering Insight:This is a standard game-loop pattern and avoids speed changes tied to monitor refresh rate.
Architecture Insight:Timing is handled once in the loop, then shared by all motion updates.
✅Behavior separated from drawing
`move()`, `handle_fleeing()`, and `draw()` do different jobs, which makes the animation easier to reason about.
Basics:One method updates motion, another computes avoidance, another paints the square.
Engineering Insight:That split makes debugging easier because the code path for each responsibility is narrow.
Architecture Insight:The project keeps a clean loop between state update and render output.
Potential Issues
⚠️Quadratic flee pass every frame
`handle_fleeing()` compares every square with every other square, so the cost grows quickly as the object count rises.
Basics:The nested loops are fine for a small demo but expensive for larger scenes.
Engineering Insight:Repeated pairwise checks create more work than the visual complexity needs.
Architecture Insight:A spatial grid or neighborhood lookup would scale better for future expansion.
⚠️Timer reset can fight fleeing
Every 4 seconds, the code forces all squares to pick a fresh random direction, which can interrupt the escape behavior mid-motion.
Basics:The square may start running away and then suddenly change course.
Engineering Insight:This can make the motion look jittery or inconsistent even when the logic is working.
Architecture Insight:Behavior rules should usually be blended, not replaced wholesale, if they are meant to coexist.

Performance
🚀 What works well
The frame draws simple rectangles and text, keeps allocations low, and uses a small number of square objects. For a classroom-sized scene, the cost stays comfortably light.
🚀 Another win
Motion is computed with plain arithmetic and `pygame.draw.rect`, so the update path is easy to understand and does not depend on heavy assets or file access.
🐢 Main risk
The fleeing pass is O(n²) because every square checks every other square each frame. That is the first thing that will dominate runtime if the object count grows.
🐢 Secondary risk
`handle_fleeing()` repeats distance and center math inside the nested loop, so it spends extra time on the same geometry data instead of caching or narrowing the search.
4 / 6
Fine for a small demo; the avoidance system is the only part that will scale poorly.
